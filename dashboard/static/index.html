DCS НЕДОСТУПЕНПочему Patroni и чтение при этом могут оставаться доступными

DCS нужен Patroni для согласования ролей, но Patroni и PostgreSQL продолжают работать на каждом узле самостоятельно. Поэтому ответ REST API ещё не означает, что узел может безопасно стать первичным сервером.

Patroni REST APIПродолжает показывать локальное состояние процесса и PostgreSQL, даже когда связь с etcd потеряна.
Первичный сервер и записьЕсли не включён DCS Failsafe Mode и Patroni не может продлить лидерскую блокировку (leader lock), он должен понизить первичный сервер или запретить на нём запись.
Реплика и чтениеРеплика может оставаться запущенной, применять полученный WAL и обслуживать запросы только на чтение. Но без DCS она не может получить право стать новым первичным сервером.

Что означает зелёный маршрут чтения: HAProxy отвечает и через Patroni API видит реплику, готовую принимать подключения. Это не подтверждает доступность DCS и не означает, что панель выполнила пробный SQL-запрос.

LEADER LOCK

Лидерская блокировка — временная запись о том, какой участник сейчас имеет право управлять первичным сервером. Patroni должен регулярно продлевать её; после истечения новое право сможет получить только один участник.

Что сопоставить: имя первичного сервера, владельца лидерской блокировки и доступность DCS. Подробный механизм переключения разобран в модуле 2.

ЕСЛИ ПЕРВИЧНЫЙ СЕРВЕР ПЕРЕСТАНЕТ РАБОТАТЬ

Как Patroni примет решение о переключении

Patroni не выбирает новый первичный сервер только потому, что прежний перестал отвечать. Узлы должны согласовать решение через DCS: сначала истекает прежнее право на лидерство, затем одна подходящая реплика получает новое.

1Блокировка не продленаDCS ждёт истечения ttl; краткий сетевой сбой ещё не означает переключение.
→
2Реплики проверяют себяКаждая Patroni сверяет состояние PostgreSQL, отставание и ограничения конфигурации.
→
3Один узел получает право стать лидеромАтомарная запись в DCS не позволяет выбрать сразу двух победителей.
→
4Только затем повышается репликаPatroni переводит PostgreSQL победившего узла в роль первичного сервера.
Почему лидерской блокировки недостаточно: защита от двух первичных серверов
FИзоляция (fencing): не допустить запись на двух серверах
сторожевой таймер не настроен

Зачем это нужно. При разрыве сети прежний первичный сервер может продолжать обслуживать открытые соединения, хотя его лидерская блокировка в DCS уже истекла. Если в этот момент повысить реплику, оба узла смогут принимать разные изменения — возникнет раздвоение кластера (split-brain).

1Потеря DCSPatroni на прежнем первичном сервере не может продлить лидерскую блокировку.
→
2СамоизоляцияPatroni должен понизить или остановить локальный PostgreSQL.
→
3Новая лидерская блокировкаТолько после освобождения права реплика получает блокировку и повышается до первичного сервера.
Что защищает сейчас

Лидерская блокировка определяет право на запись, а работающий Patroni выполняет самоизоляцию (self-fencing): останавливает или понижает PostgreSQL при потере DCS.

Граница этого стенда

Сторожевой таймер (watchdog) не настроен. Если Patroni или вся ОС зависнет и не сможет выполнить самоизоляцию, стенд не гарантирует остановку прежнего первичного сервера.

Что не обеспечивает изоляцию

HAProxy перестанет направлять новые соединения на прежний первичный сервер, но не остановит PostgreSQL и не гарантирует разрыв уже открытых соединений.

Правильный порядок: сначала гарантированно запретить запись на прежнем первичном сервере, затем повышать реплику. В промышленной среде для этого используют сторожевой таймер (watchdog) или внешнее отключение узла (STONITH).

Стабильные адреса для приложения

Роли db1 и db2 могут поменяться, но приложение продолжает подключаться к одному адресу. HAProxy проверяет Patroni и направляет каждое новое соединение на узел подходящей роли. Уже открытое соединение он не переносит.

Как выбирать маршрут:5000 — запросы, которые могут изменять данные. :5001 — отдельная нагрузка только на чтение; на реплике изменения могут появиться немного позже, чем на первичном сервере.

Зачем это происходит

Если первичный сервер станет недоступен, Patroni сможет повысить достаточно актуальную реплику. Репликация уменьшает время восстановления, но не заменяет резервную копию.

WAL ещё не получен — отставание при получении (receive lag); причина не обязательно в сети WAL ещё не применён — очередь применения (replay queue) на самой реплике Полное отставание — replay lag относительно первичного сервера

Как выполнить плановое переключение (switchover) в реальной эксплуатации

Обычно используют patronictl: команда проверит состояние кластера и обратится к Patroni. Без --force она запросит подтверждение.

patronictl …

Ту же операцию можно запросить через REST API текущего первичного сервера:

curl …
После команды проверьте новый первичный сервер, доступность записи, состояние прежнего первичного сервера и отставание репликации. Код HTTP 200 или 202 означает, что запрос принят, но не подтверждает завершение переключения.

Как выполнить перезапуск или перечитать конфигурацию в реальной эксплуатации

Перезапуск (restart) выбранного PostgreSQL под управлением Patroni:

patronictl … restart
curl … /restart

Перечитывание локальной конфигурации Patroni (reload):

patronictl … reload
curl … /reload
В рабочей среде узлы обслуживают по одному, обычно начиная с реплик. Команда systemctl restart postgresql действует в обход Patroni, поэтому для штатного обслуживания её использовать не следует.

Как проводят такие проверки в рабочей среде

В рабочей среде отказ воспроизводят только по согласованному плану и в отведённое окно работ. Способ зависит от платформы: управляемое прерывание работы в Kubernetes, остановка виртуальной машины или системной службы, либо контролируемая сетевая изоляция.

До испытания определяют допустимое время восстановления (RTO), допустимую потерю данных (RPO), способ наблюдения за приложением и порядок аварийного восстановления. Команды docker compose stop … выше предназначены только для этого учебного стенда.

РЕЖИМ РУЧНОГО ОБСЛУЖИВАНИЯ · PAUSE / RESUME

Временно ограничить автоматические действия Patroni

ответственность переходит администратору
Зачем нужен режим обслуживания (pause)

Он предназначен для восстановления после повреждений, обновления основной версии или сложной ручной перестройки. Patroni продолжает публиковать состояние и продлевать лидерскую блокировку, но не запускает остановленный узел и не повышает реплику автоматически.

Применяйте: только когда намеренно управляете PostgreSQL вручную.

Не применяйте: для обычного перезапуска, перечитывания конфигурации или планового переключения.

Риск: Patroni может предупредить о двух первичных серверах, но не обязан устранить это состояние. Перед возобновлением автоматического управления (resume) убедитесь, что первичный сервер только один.

Как включить и выключить режим обслуживания в реальной эксплуатации

Через patronictl с ожиданием всех участников:

patronictl … pause --wait
patronictl … resume --wait

Тот же запрос через REST API любого доступного Patroni:

curl … pause=true
curl … pause=false
Признак pause хранится в DCS и действует на весь кластер. После работ убедитесь, что первичный сервер только один, и выполните resume.

ДОЛГОВРЕМЕННАЯ ИСТОРИЯ POSTGRESQL

История линий времени (timeline)

После повышения реплики PostgreSQL создаёт новую линию времени (timeline) — отдельную ветвь истории WAL. Этот список помогает связать смену роли с началом новой ветви.

Как использовать эту историю при расследовании
  1. Найдите время появления новой линии: примерно тогда реплика была повышена до первичного сервера.
  2. Сопоставьте его со сменой лидера и недоступностью записи на графике слева.
  3. Проверьте, вернулся ли прежний первичный сервер как реплика и находится ли он на текущей линии времени. Другая линия может означать расхождение истории WAL.

Это не журнал SQL-запросов и не аудит данных. Он отвечает на вопрос: «когда кластер начал новую ветвь WAL и в какой точке?»
